# Fontes/APIs de tradução "de dicionário" (sem IA) EN→PT-BR para um leitor: pesquisa comparativa e recomendação

## TL;DR
- Para o caminho "tradução comum" ao selecionar uma palavra/expressão num leitor comercial, a combinação vencedora é **dicionário primeiro, MT como fallback**: use dados do **Wiktionary via dumps do Kaikki.org (wiktextract)** self-hosted como fonte de verbete (grátis, cacheável, licença CC BY-SA com atribuição), e o **Google Cloud Translation API (Basic/v2, US$20 por milhão de caracteres, 500 mil grátis/mês permanentes e compartilhados entre v2 e v3)** como fallback de MT — oficial, estável e com cache legal por até 15 dias.
- **DeepL** entrega a melhor qualidade em prosa europeia, mas é o mais caro (US$5,49/mês + US$25/milhão), cobre PT-BR como variante própria e **não é superior em phrasal verbs/idiomatismos** (num teste da professora de idiomas Jessica Santero, na WhistleOut, o DeepL não acertou "um único" idiomatismo, enquanto Google e Microsoft acertaram); **Microsoft/Azure** é o mais barato entre os oficiais (US$10/milhão, 2 milhões grátis/mês).
- Evite como fontes primárias em produto pago: **Lingva Translate** e outros proxies não-oficiais do Google (violam o ToS do Google e são instáveis), **Linguee/WordReference** (sem API oficial; scraping proibido), **Glosbe** (API oficial desativada por ataque DDoS) e **MyMemory** (limite diário baixo e qualidade inconsistente).

## Key Findings

### O caso de uso favorece "dicionário primeiro, MT depois"
Ao selecionar **uma palavra isolada**, o usuário se beneficia de um verbete (múltiplos sentidos, classe gramatical, exemplos) — algo que só uma fonte de dicionário oferece; um endpoint de MT devolve apenas uma tradução única e descontextualizada. Já para **frases curtas** e muitos **phrasal verbs**, a MT costuma se sair melhor porque produz uma tradução fluida mesmo quando não há verbete. A arquitetura ideal é: (1) tentar o verbete de dicionário; (2) se não houver entrada, cair para MT. Isso maximiza qualidade e minimiza custo, porque o dicionário local é grátis e cacheável, e o MT pago só é acionado nas lacunas.

### Tradução automática (MT) — APIs oficiais
- **Google Cloud Translation (Basic v2 / Advanced v3):** US$20 por milhão de caracteres (ambas edições), com **500.000 caracteres/mês grátis permanentes** — a página oficial confirma que o crédito é compartilhado entre v2 e v3 ("Credit usage applies collectively to both Cloud Translation - Basic and Cloud Translation - Advanced") e não expira. Exige projeto no Google Cloud e chave/conta de faturamento. Cobre EN→PT-BR (código pt-BR). Cache: os termos permitem armazenar resultados em cache temporário por até **15 dias** por ação iniciada pelo usuário (versão histórica do ToS AJAX); a documentação atual da Cloud Translation afirma que o texto é mantido apenas brevemente em memória e o Google **não usa o conteúdo para treinar** seus modelos. Uso comercial permitido. Confiabilidade alta (serviço oficial, SLA do Google Cloud). Exemplo de custo: 10 milhões de caracteres ≈ US$190/mês.
- **DeepL API (Free e Pro):** Free = **500.000 caracteres/mês grátis** (chave termina em `:fx`, endpoint `api-free.deepl.com`); Pro = **US$5,49/mês + US$25 por milhão de caracteres**, sem desconto por volume. Cobre **Português (Brasileiro) e (Europeu)** como variantes distintas (PT-BR/PT-PT), com parâmetro `formality` suportado em português. Qualidade reconhecida como superior para línguas europeias em prosa, mas testes independentes mostram que **DeepL traduz idiomatismos/phrasal verbs pior que Google e Microsoft**. Cache/privacidade: no Free/Developer, os termos reservam a DeepL o direito de **armazenar perpetuamente** o conteúdo e usá-lo para melhorar os modelos; no **Pro** os textos **nunca são usados para treino e são apagados imediatamente** após a tradução. Você mesmo constrói sua camada de cache (DeepL não faz cache). Não permite chamadas diretas do browser (erro CORS 403). Uso comercial permitido no Pro.
- **Microsoft Azure AI Translator:** tier grátis **F0 = 2 milhões de caracteres/mês** (não expira, um por conta; cobre text translation, detecção de idioma, dicionário bilíngue e transliteração); padrão **S1 = US$10 por milhão de caracteres** (o mais barato entre os oficiais). Exige recurso Azure e chave. Cobre 100+ idiomas incluindo PT-BR. Qualidade de idiomatismos considerada a mais fraca dos três grandes por algumas avaliações, embora um teste tenha apontado Microsoft acertando idiomatismos melhor que DeepL. Uso comercial permitido.
- **Yandex Translate API (Yandex Cloud):** preço por caractere, **sem tier grátis fixo**; cerca de US$4,10 por milhão de caracteres na região da Rússia (sem IVA); reportado em torno de US$15/milhão em outras faixas. Suporta ~90 idiomas incluindo PT. Uso comercial permitido, mas há preocupações de disponibilidade/faturamento fora da Rússia e a qualidade EN-PT é considerada inferior à do Google/DeepL. Menos recomendável para PT-BR.

### Tradução automática — opções open-source / não-oficiais
- **LibreTranslate:** open-source (licença **AGPLv3**), engine Argos Translate. Self-hosted = sem limites e privado, mas exige infraestrutura (modelos de vários GB, GPU para desempenho aceitável) e a **qualidade EN-PT é notavelmente inferior** às comerciais. Instância pública oficial (libretranslate.com) exige chave paga (a partir de US$9/mês, limite de 80 req/min); os termos dizem que a API pública deve ser usada só para teste/uso pessoal/infrequente, e produção deve obter chave. Uso comercial permitido (você hospeda). Cache: livre (dados seus).
- **Lingva Translate:** front-end/proxy não-oficial que faz **scraping do Google Translate** (via Lingva Scraper). Grátis e open-source (AGPL), mas **usar resultados do Google Translate por meios não-oficiais viola o ToS do Google**; instâncias públicas são instáveis e podem cair/ser bloqueadas. **Não recomendado para produto comercial** — risco legal e de confiabilidade.

### Fontes de DICIONÁRIO (verbetes: definições + traduções)
- **Wiktionary / Wikcionário:** conteúdo dual-licenciado **CC BY-SA 4.0 + GFDL** (uso comercial permitido com **atribuição obrigatória + share-alike**). Melhor acesso para produção: **dumps do Kaikki.org gerados por wiktextract** (mantido por Tatu Ylönen) em JSONL — o dump bruto `raw-wiktextract-data.jsonl` tem **23,1 GB (2,6 GB comprimido em .gz)**, atualizado ao menos semanalmente (versão verificada extraída em 2026-07-09 do dump enwiktionary de 2026-07-06). Cada entrada traz um campo `translations` com `lang`, `code` (ex.: `"pt"`), `word`, `sense` e `tags`. **Limitação PT-BR crítica:** o wiktextract usa apenas `code:"pt"` para todo o português; PT-BR precisa ser inferido via `tags`/categorias ("Brazil"/"Brazilian Portuguese"), e as tabelas de tradução EN→PT geralmente **não** distinguem a variante. Cobertura de phrasal verbs/idiomatismos existe (entradas dedicadas como "give up") mas é incompleta e nem sempre desambiguada por sentido. A API oficial da Wikimedia (MediaWiki Action/REST) tem rate limits e **exige User-Agent significativo**; a própria Wikimedia recomenda **dumps em vez de chamadas ao vivo** para alto volume. Para reuso live/SLA há o **Wikimedia Enterprise** (tier grátis com 50.000 req/mês on-demand; contrato com SLA de até 99%).
- **Free Dictionary API (freedictionaryapi.com, sucessor de dictionaryapi.dev):** grátis, **sem chave**, **1.000 requisições/hora por IP**. Conteúdo do **Wiktionary sob CC BY-SA 4.0** — exige atribuição visível e link. Tem campo opcional `translations` e suporta múltiplos idiomas via código na URL, mas a **cobertura de definições é primariamente em inglês** (o dictionaryapi.dev original passou a usar Wiktionary com foco em inglês). Serviço community-run, sem SLA — confiabilidade moderada; cacheável e recomendado cachear.
- **Linguee:** **sem API oficial pública**; os termos proíbem repassar os serviços a terceiros mediante pagamento. Wrappers não-oficiais (linguee-api) fazem scraping e ficam sujeitos aos termos. **Não recomendado** para uso comercial.
- **WordReference:** **não tem mais API oficial**; nos fóruns oficiais a política é que apps que fazem scraping "provavelmente serão bloqueados". **Não recomendado.**
- **Glosbe:** tinha API gratuita (dados agregados de fontes abertas como Wiktionary/TM), historicamente citada como a única compatível com software livre; porém a **API está atualmente desativada** ("Due to the long lasting DDoS attack ... we have to shut down the service"). Inviável como dependência de produção no momento.
- **DeepL — glossário vs. tradução:** o glossário do DeepL **não é um dicionário**; é um recurso para forçar termos específicos (marcas, jargões) dentro do endpoint de tradução. Não serve como fonte de verbete/definições.
- **MyMemory (Translated):** memória de tradução (dados humanos + ModernMT como fallback). Grátis sem chave, mas **limite diário baixo**: documentação oficial diz "Free, anonymous usage is limited to 5000 chars/day. Provide a valid email ('de' parameter)... and enjoy 50000 chars/day. Are you a CAT tool maker? Get whitelisted! Write us and get 150000 chars/day!". Qualidade inconsistente (depende de haver segmento similar na memória). Cobre EN↔PT-BR. Termos exigem atenção para uso comercial em escala (planos pagos via RapidAPI, ~US$22/milhão). Bom para protótipo, ruim para produção séria.

### Estimativa de custo em escala (app comercial com muitos usuários)
Como o alvo são palavras/expressões curtas (poucos caracteres por seleção) e o resultado é altamente cacheável, o volume pago efetivo tende a ser modesto. Exemplo, para 10 milhões de caracteres/mês de MT (após descontar o cache e o dicionário local): ≈ **US$190/mês no Google** (10M − 0,5M grátis × US$20/milhão), **US$100/mês no Azure**, e **≈US$255/mês no DeepL Pro** (US$5,49 + 10M × US$25/milhão). Um cache agressivo (o mesmo termo raramente muda) reduz drasticamente esses números ao longo do tempo.

## Recommendations

1. **Arquitetura recomendada — dicionário local + MT de fallback, com cache agressivo:**
   - **Fonte primária (verbete):** monte um dicionário EN→PT-BR offline a partir dos **dumps do Kaikki.org (wiktextract)**, indexado localmente (SQLite/Elasticsearch). Cumpra a licença CC BY-SA exibindo atribuição visível ("definições do Wiktionary, CC BY-SA") e link de volta. Isso dá custo marginal zero por consulta, latência baixa e verbetes ricos para palavras isoladas.
   - **Fonte de fallback (MT):** **Google Cloud Translation Basic (v2)** para quando não houver verbete (frases curtas, phrasal verbs sem entrada, gírias). É oficial, estável, cobre PT-BR, tem 500 mil caracteres/mês grátis e permite cache. Faça cache no seu backend (chave = hash do texto-fonte) e revalide periodicamente para respeitar o espírito dos termos.

2. **Se qualidade de prosa for prioridade e o orçamento permitir:** troque o fallback por **DeepL Pro** — mas saiba que ele **não** é a melhor escolha para phrasal verbs/idiomatismos e é o mais caro. Uma alternativa de baixo custo é **Azure Translator (US$10/milhão, 2 milhões grátis)** como fallback, que ainda traz dicionário bilíngue embutido útil para palavras isoladas.

3. **Não use em produção:** Lingva/proxies não-oficiais do Google (risco de ToS e instabilidade), Linguee e WordReference (sem API oficial; scraping proibido/bloqueado) e Glosbe enquanto a API estiver desativada. **MyMemory** só para protótipo.

4. **Cumpra as licenças:** para qualquer dado de Wiktionary (direto, via Kaikki ou via Free Dictionary API) mantenha atribuição + share-alike; valide com assessoria jurídica o efeito do share-alike sobre a base de dados derivada incorporada ao seu produto pago.

### Benchmarks que mudariam a recomendação
- Se o volume pago de MT ultrapassar ~50 milhões de caracteres/mês mesmo após cache, reavalie migrar o fallback para **Azure (US$10/milhão)** ou negociar volume.
- Se a cobertura de PT-BR do Wiktionary se mostrar insuficiente em testes com seu corpus de leitura (meça a taxa de "verbete encontrado"), aumente o peso do fallback MT ou adicione um dicionário bilíngue licenciado.
- Se precisar de dados de Wiktionary ao vivo em alto volume, migre para **Wikimedia Enterprise** (SLA) em vez de bater na API pública.

## Caveats
- Preços e limites de free tier mudam com frequência; os valores aqui refletem o material público disponível em julho de 2026 e devem ser confirmados nas páginas oficiais antes de decisões de orçamento. Vários números de preço vêm de blogs terceiros (ex.: Langbly, Costgoat) e não das páginas oficiais.
- Os termos de cache do Google citados (15 dias) vêm de uma versão histórica do ToS da antiga API AJAX; confirme a cláusula de cache aplicável à Cloud Translation atual no contrato vigente.
- A distinção PT-BR vs PT-PT no Wiktionary é frágil (código único "pt"); não assuma variante brasileira sem checar tags.
- Rate limits exatos da API pública da Wikimedia e alguns números de Yandex/DeepL não são publicados de forma estável e podem variar por conta/região.
- A qualidade relativa de idiomatismos entre Google, DeepL e Microsoft varia por teste e por par de idiomas; trate as afirmações de "melhor/pior" como tendências, não verdades absolutas.
